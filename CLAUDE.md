# ImageSpell 開発ルール(CLAUDE.md)

このファイルは全Claude Codeセッションが毎回読む恒久ルール。
経緯・進捗・状態はREADMEに、恒久ルールはここに書く。

## このリポジトリについて
- imagespell.com — 英語圏向けブラウザ完結型画像ツール群のモノレポ
- Cloudflare Pages配信。public/ がビルド出力ルート
- 規約: 1ツール=1ディレクトリ(public/<tool-name>/)。ツールは今後も増える
- 全ツール共通の売り: 完全クライアントサイド処理(画像をサーバーに送らない)。
  この原則を破る実装は提案段階で人間に確認

## No-go(常に禁止。明示的な指示があっても実行前に人間へ再確認)
- URL構造の変更(/pixelate-image, /depth-map-generator 等)。
  リダイレクトによる「整理」も禁止(SEO資産に直結)
- public/depth-map-generator/js/depth-config.js の値の変更(品質チューニング済み)
- 深度モデルの変更。Depth Anything V2 **Small** 固定
  (Base/Large は CC-BY-NC 非商用ライセンスのため使用禁止)
- Lemon Squeezy の設定値・License API ロジック・?purchase=success 処理の変更
- .env の内容変更・コミット(.gitignore 対象であることを常に維持)
- APIキー・シークレットをフロントエンドコード・コミット・ログに含めること
- 公開中のOG画像・LPテキストの変更(明示的な指示がある場合のみ可)

## 素材ルール(画像を扱うとき必ず)
- 公開画像は「人が写っていない・依頼者の自前撮影・公共スペースで撮影」のみ
- 公開前にEXIF(特にGPS)を除去する。scripts/strip-exif.sh を使用。
  注意: macプレビューの「書き出す」ではGPSが消えないことが確認済み
- ファイル名・alt属性・キャプションに撮影地名を入れない
  (汎用的な英語の内容記述のみ。例: park-with-pine-grove)
- 画像に関する指示・報告では、ファイル名だけでなく内容の言語化を併記する
  (取り違え防止)
- 需要があっても、権利侵害・悪用が主用途になり得る機能は作らない
  (例: 他者のwatermark除去)。該当しそうなときは人間に確認

## 作業様式
- ブランチ: feat/<tool-name>。自分の担当ディレクトリ(public/<tool-name>/)の
  外は変更しない
- 共有ファイル(ルートの index.html、sitemap.xml、共通CSS/ヘッダー部品)は
  「統合作業」として明示的に指示された場合のみ変更する。並行セッションとの
  衝突源のため
- コミット: 1論点=1コミット。conventional commits 形式
  (feat: / fix: / docs: / refactor: ...)
- CSS/JS を更新したらキャッシュバスティング(?v=N)の更新を忘れない
- 新ツールの canonical・sitemap・内部リンクは末尾スラッシュあり
  (`https://imagespell.com/<tool-name>/`。Cloudflare Pages の配信実態と一致する形)で
  宣言する。公開前に `curl -I` で canonical 記載URLが 200 を返すことを確認する
  (スラッシュなしは 308 になり、GSC でリダイレクトエラーの原因になる)
- モバイルは実機タップ前提で作る: input/フォーム要素の font-size は 16px 以上
  (iOS Safari の自動ズーム対策)、ファイル選択は <label>+<input type="file">
  の素直な関連付け
- 新ツールのページは既存ツールと同型のSEO構成を踏襲:
  title / meta description / OGP / canonical / How-to / ユースケース /
  FAQ + JSON-LD / ホームと相互リンク / sitemap.xml 更新(統合作業時)

## エスカレーション基準
- 自力で解決してよい: 実装バグ、ライブラリ選定、エラー対応
- 同じ問題に3回失敗したら、人間に聞く前に /model で上位モデル
  (利用可能なら fable、なければ opus)に切り替えて再挑戦してよい。
  解決したら元のモデルに戻す(コスト管理のため)
- 人間に確認してから進める(モデルを上げても省略不可):
  仕様の変更を伴う判断/価格・課金・法務・権利に関わること/
  UXのトレードオフで方針が割れるとき/No-goに触れそうなとき
- 人間に質問するときは、選択肢を2〜3個に絞り、推奨案を1つ添える

## バグ修正指示の受け方
- 指示は「現象・期待する状態・完了条件」の3点セットで来る
- 完了条件をそのまま検証してから完了報告する。自分の解釈による別の
  検証項目にすり替えない(過去に「カード内部の整列」を直して
  「カードのページ内配置」の指摘を見落とした事故がある)
- 完了条件が曖昧な指示を受けたら、着手前に完了条件を確認する
